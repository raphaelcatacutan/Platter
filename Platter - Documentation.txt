Overview of the Programming Language
Platter is a beginner-friendly programming language designed to help learners grasp the fundamentals of coding through the familiar lens of culinary arts. Developed by the group DeServe, Platter draws inspiration from the concept of mise en place, the practice of organizing ingredients before cooking, to promote clarity, structure, and readiness in programming. Reserved words and syntax are modeled after kitchen terminology, making abstract concepts more intuitive and relatable. Instead of traditional code blocks, Platter uses the term platter, visually and conceptually echoing the {} braces that contain logic, much like a real-life platter holding a complete dish.
Platter’s syntax and semantics are inspired by a blend of established programming languages: it adopts the program structure, identifiers, data types, declarations, structs, and comments of C; the operators and typecasting of Python; and the statements and methods of Java, without the complexity of object-oriented design. This hybrid approach allows learners to build transferable skills while staying grounded in a simplified, modular environment. Recipes in Platter serve as reusable units of logic, akin to subprograms, helping learners write clean, maintainable code with a focus on task-specific clarity.
To reinforce discipline and design awareness, Platter enforces a 25-character limit on identifiers, inspired by the 25% plating criterion in culinary competitions. This constraint encourages concise naming and thoughtful structure, mirroring the precision required in both programming and presentation. With its culinary metaphors, modular design, and pedagogical clarity, Platter offers a fresh and engaging way to learn programming serving both as a teaching tool and a stepping stone toward deeper technical fluency.

2. General Rules
1. The program structure order should follow the order of global declarations, recipe declarations, and start recipe.
2. The program executes from top to bottom, unless a control structure changes the flow of execution.
3. Identifiers are case sensitive and must only be composed of alphanumeric characters (a to z, A to Z, 0 to 9) and underscores ( _ ), not start with any number, no other special characters and spaces, and have a limit of  25 characters.
4. All global declarations, including ingredients, arrays, table prototypes and instances, and recipe declarations cannot use the same identifier regardless of their differences in data types and serve types.
5. Local declarations (ingredients, arrays, tables, spices, recipe calls) cannot share the same identifier within a scope. Each recipe, including the start recipe, and each platter has its own local scope. Recipe identifiers belong to the global scope, while spices and platter identifiers belong to the recipe’s local scope.
6. Identifiers declared in a broader scope but not in the current local scope can be accessed. This way, changes made in the local scope affect the same identifier in the broader scope. Nested platters in loops or conditionals create deeper local scopes with their own declarations and a separate set of identifiers.
7. The primitive data types refer to the following: piece, sip, chars, and flag, while collections refers to arrays and tables.
8. The literals are referred to the following: (a) numeric: piece and sip literals, (b) textual: chars literal, and (c) boolean: flag literal.
9. Ingredients are the variables, arrays are the arrays, and tables are the structs.
10. Ingredients are containers of primitive data types that can be accessed using its assigned identifier and must be declared with a specific data type using the ( of ) keyword, and if initialized, the assigned value must be compatible with the declared type. 
11. All uninitialized ingredients are assigned a default value based on their data type.
12. Null and void data types are not supported since all data types have their own default value when uninitialized. 
13. Values are units of usable data. Categorized into the following: (a) Expressions and (b) Notation Value. Notation value can either be Array Notation Value or Table Notation Value.
14. Values can be referred to as chars_value, numeric_value: piece_value and sip_value, or boolean_value referring to all values that are expected to resolve to those data types.
15. Operands are the resolved sources of values used in expressions. It may originate from the following sources: (a) Chars literals, (b) Flag literals, (c) Numeric literals (piece and sip literals), (d) Chars Ingredient, (e) Flag Ingredient, (f) Numeric ingredient (piece and sip ingredient), (g) Recipe calls of primitive serve type (piece, sip, flag, chars), (h) Accessed array index of a single dimension array, (i) Accessed field of primitive data types in a table, and (j) Valid Expression.
16. Expressions can be defined as arithmetic, relational, logical, and complex expressions.
17. There are also assignment and compound assignment operators.
18. Arithmetic expressions support up to 15 digits of precision for discrete values and up to 15 digits before and 7 digits after the decimal point for continuous values.
19. When writing an expression, no two operands may appear consecutively without an operator.
20. Prefix unary operators are supported, no postfix unary operators are allowed.
21. Nesting expressions are allowed, wherein parentheses should be used, and they must be balanced.
22. Nested expressions are evaluated from the innermost to outermost pair of parentheses.
23. Evaluating expressions follow an order of precedence, and expressions of the same precedence are evaluated from left to right.
24. Arithmetic, Relational, and Logical expressions and Recipe calls may be combined to create complex expressions.
25. All expressions must resolve to values within the valid range of their declared data type. If an expression produces a value outside this range, the program halts and reports the originating recipe or statement that caused a value beyond the data type’s range as part of the error. This rule applies to all arithmetic, assignment, and serve operations, ensuring semantic integrity and preventing overflow and invalid state propagation.
26. All global declarations may only consist of the following: ingredient, arrays, table instance and table prototype
27. All global declarations must be declared prior to the recipes and the start recipe.
28. Ingredients, arrays and tables must be declared before they are used in statements.
29. No assigning or reassigning can be done within the global scope as no other statements apart from declarations are allowed.
30. Globally declared ingredients, arrays, table instances and prototypes are visible throughout the entire program, across all recipes, the start recipe, control-flow statements, platters, arrays, and tables. 
31. A program is not required to have global declarations.
32. An array is a collection of values of the same type that can be organized in multiple dimensions.
33. An array notation represents a list of elements enclosed by a pair of square brackets and separated by a comma ( , ).
34. An array accessor which uses a piece as an index to the element within each pair of brackets following the array’s identifier, is used to read and write a value to an array.
35. A table prototype defines the fields for a table instance and is used as the template for table declaration.
36. A table instance is a collection of fields defined by a table prototype.
37. A table instance notation represents a list of fields and its values enclosed by a pair of square brackets ( [ ] ).
38. A table accessor which uses a colon and the field name following the table’s identifier, is used to read and write a value to a table.
39. An array or table may contain pieces, sips, chars, flags, nested arrays, or nested tables.
40. Values in an array or table must match the type specified in its declaration.
41. Operators cannot be applied directly between arrays or tables.
42. A platter is a block of executable statements enclosed in open and close curly brackets ({ }). 
43. Every executable statement inside a platter does not require to be indented.
44. A platter defines a local scope.
45. Every statement that is associated with a platter is assigned its own distinct instance of that platter, including recipes and the start recipe.
46. Recipes are user-defined subprograms that should be defined after the global declarations, if there are any, and before the start recipe.
47. All recipes must be defined first before it is called.
48. There can be zero or many user-defined recipes in a program, in which there can be no recipes defined within the start recipe or another recipe.
49. Recipe parameters are referred as spices, while passed arguments are referred as flavors for recipe calls.
50. When defining recipes, it must include its serve type, keyword ( of ), identifier, spice list, which may be empty, and a platter, with at least one reachable serve statement.
51. A recipe may have many or not have any local declarations and depend on available global declarations for its ingredients, arrays, or tables. 
52. Return statements are referred to as serve statements.
53. Recipes can only have one serve type, in which it can either be piece, sip, flag,  chars, arrays, or tables. 
54. Each recipe must contain exactly one top-level, defined at the program’s outermost scope,  platter. 
55. Nested recipe definitions are not allowed.
56. Recipes only allow passing by value, no addresses or container references are passed around as flavors and each recipe serves a value as its output.
57. A recipe can call any other recipe that was defined before it.
58. Recipes cannot be called globally or anywhere outside a local scope.
59. A recipe calling itself within its own platter is allowed.
60. A recipe call can be used as a standalone statement and must end with a statement terminator (;). In this form, the recipe is invoked for its behavior, and its served value is discarded unless explicitly handled.
61. User-defined recipes cannot have the same identifiers as built-in recipes.
62. Calling built-in recipes adhere to the same rules that apply to user-defined recipes.
63. Built-in recipes are classified as the following: (a) Type Casting: topiece, tosip, and tochars, (b) Input and Output: take and bill, (c) Math, Formatting, and Random: pow, sqrt, fact, cut, copy, and rand, and (d) Arrays and Tables: size, sort, search, reverse, append, remove, and matches.
64. A standalone take recipe call is referred to as the input statement and a standalone bill recipe call is referred to as the output statement.
65. Local declarations may contain ingredient, array, or table declarations.
66. Local declarations are only visible within the platter or recipe in which it is defined. The scope of a local declaration begins at its point of declaration and ends at the closing curly bracket of the platter where it was declared.
67. All local declarations can only be declared within a platter.
68. Local declarations are not required within a platter.
69. The following are defined as statements: (a) local assignment, (b) standalone recipe call, (c) conditional: check and menu, (d) looping: pass, repeat, and order-repeat, and (e) jump: next, stop, and serve. Although they are technically statements, global and local declarations are not included in our definition of statements for clear distinction and separation in the program structure and syntax. Initialization is considered a part of declaration and assigning of values without a declaration will be considered as a local assignment statement.
70. The semicolon ( ; ) is used as a statement terminator and it is only used to terminate the following: (a) declaration: global and local, (b) local assignment, (c) standalone recipe calls, (e) pass: initialization and update, and (e) order-repeat.
71. The control-flow statements refers to the following: check as the if statement, menu as the switch statement, pass as the for statement, repeat as the while statement, order-repeat as the do-while statement.
72. Conditional, looping, and jump statements cannot be created in the global scope. They are also not considered as recipe calls as they do not serve any value and cannot be used in expressions.
73. All control-flow statements are paired with a platter, following the platter rules.
74. All control-flow statements can only be used within user-defined recipes or within the start recipe. 
75. All control-flow statements can read, modify, or reference an ingredient outside its platter scope.
76. Nested control-flow statements are allowed. They can be nested inside another control-flow statement, including one of the same type.
77. The next statement and stop statement can be placed in nested loops and only the innermost loop where it is placed will be affected.
78. There must be exactly one start recipe in a program. 
79. The start recipe is defined after the global ingredients, and recipe declarations. 
80. The start must have an empty spice list. 
81. The start recipe cannot be invoked in any part of the program.
82. A comment is not read by the compiler when the program is compiled. 
83. Comments can be added anywhere where it will not interrupt a specific reserved word or identifier by being added between it.
84. A comment can contain any number of characters, including alphanumeric and special characters.
85. A Multi-line comment begins with a closing double hash (##) and ends when another ( ## ) is encountered.

	3. Specific Rules
Identifiers
Identifiers are used to name user-defined ingredients, recipes, arrays, spices, and tables. Identifiers enable access and manipulation of these elements in the program by providing a way to reference them.


Rules:
1. May only contain lower (a to z) and upper (A to Z) case English letters, digits (0 to 9), and underscores ( _ ). All other special characters are not allowed.
2. Must begin with at least one letter (A to Z, a to z) or an underscore.
3. Cannot begin with a digit.
4. Must only be made of a maximum of 25 characters.
5. Cannot contain whitespaces anywhere between the characters.
6. Case sensitive where the upper and lower case letters are considered different from each other and creates a completely different identifier.
7. Cannot match any reserved word.

Data Types and Literals
Data Types
Data types are used to define the type of value stored by ingredients, arrays, and spices, as well as the serve types of recipes. The primitive data types in Platter consist of piece, sip, flag, and chars. Data types in Platter are all immutable, meaning a data type cannot be changed once declared on an ingredient or array, and statically and explicitly typed.


Literals
Literals are the values themselves that are stored in ingredients or used in different sections of the program. Platter categorizes its literals as numeric: piece and sip literals; textual: chars literal; and boolean: flag literal.




piece
A data type for storing 15-digit signed integers.
Rules for piece:
1. piece ingredients can only store up to 15-digit signed integers and will not allow assigning of values beyond this nor will it allow assigning of a sip, flag, or chars data type or literal.
2. Defaults to 0 if uninitialized.


Rules in defining a piece literal:
1. piece values must only be an integer.
2. piece values cannot include fractions or decimal points.
3. piece values cannot contain whitespaces or any other non-digit characters in between the digits.
4. piece values cannot contain any other non-digit character before, aside from the unary minus (-), and after the digits.
5. piece values must remain within the range of -999999999999999 to 999999999999999. Any value that exceeds this range will not be allowed.
6. piece values may either be positive, negative, or zero values.
7. piece values cannot have leading zeros.
8. piece values that are unsigned (without the negative sign) are positive.
9. piece values should not have the unary plus (+) sign as a prefix of positive values.
10. piece values that have the unary minus (-) sign as prefix are considered as negative values.
11. A piece with a value of 0 can have a unary minus (-) prefix (-0) but will resolve to an unsigned 0 if created.

sip 
A data type for storing up to 15 signed non-fractional digits and 7 fractional digits separated by a decimal point (.).


Rules for sip:
1. sip ingredients can only store up to  15 signed non-fractional digits and 7 fractional digits separated by a decimal point (.), and will not allow assigning of values beyond this nor will it allow assigning of a piece, flag, or chars data type or literal.        
2. Defaults to 0.0 if uninitialized.


Rules in defining a sip literal:
1. sip values must have a single decimal point.
2. sip values must have at least a single digit (0 to 9) before and after the decimal point.
3. sip values cannot contain whitespaces or any other non-digit characters in between the digits aside from the decimal point.
4. sip values cannot contain any other non-digit character before, aside from the unary minus (-), and after the digits.
5. sip values must remain within the range of -999999999999999.9999999face to 999999999999999.9999999. Any value that exceeds this range will not be allowed.
6. sip values cannot have leading zeros in the non-fractional digits, unless the non-fractional value is just 0.
7. sip values can have trailing zeros within the digit range limit.
8. sip values cannot be in exponential or scientific notation
9. sip values may either be positive, negative, or zero values.
10. sip values that are unsigned (without the negative sign) are positive.
11. sip values should not have the unary plus (+) sign as a prefix of positive values.
12. sip values that have the unary minus (-) sign as prefix are considered as negative values.
13. A sip with a value of zero for both the fractional and non-fractional digits (0.0) can have the unary minus (-) as a prefix (-0.0) but will resolve to 0.0 if created.

chars 
A data type for storing an empty or a sequence of ascii encoded characters. 


Rules for chars:
1. chars ingredients can only store  an empty or a sequence of ascii encoded characters enclosed in double quotes (“”), and will not allow assigning of values beyond this nor will it allow assigning of a piece, sip, or flag data type or literal.
2. Defaults to “” if uninitialized.


Rules in defining a chars literal:
1. chars values are case-sensitive.
2. chars values are enclosed between double quotes (“”).
3. chars values can be a digit.

5. chars values can use escape sequences to have access to specific symbols within a string or apply print formatting, using a backslash (\) for a non-recognized escape sequence will just print the backslash (\) together with the symbol that follows.

Operators and Expressions
Values 
Values are units of usable data. Categorized into the following:
1. Expressions
2. Notation Value 
   1. Array Notation
   2. Table Notation


Operands 
Operands are the resolved sources of values used in expressions. They may only originate from any of the following sources:
   1. Chars literals
   2. Flag literals
   3. Numeric literals (piece and sip literals)
   4. Chars Ingredient
   5. Flag Ingredient
   6. Numeric ingredient (piece and sip ingredient)
   7. Recipe calls of primitive serve type (piece, sip, flag, chars)
   8. Accessed array index of a single dimension array
   9. Accessed field of primitive data types in a table
   10. Valid Expression
 


Operators 
        Operators are the built-in symbols or keywords that are used with operands and produce a result. These include:
1. Unary Operators ( -, not )
2. Arithmetic Operators ( +, -, *, /, % )
3. Logical Operators (and, or, not)
4. Relational Operators ( >, <, ==, !=, >=, <= )
5. Assignment Operator ( = )
6. Compound Assignment Operators ( +=, -=, *=, /=, %= )








Expressions 
Expressions are combinations of values and operators or a single value that evaluates to a single result value. They ensure calculations, comparisons, decisions, and text operations can be written clearly and consistently.


Rules:
1. An expression must consist of either a single operand or a valid sequence of operands and operators, where each operator connects exactly two operands (except for unary operators).
2. No two operands may appear consecutively in an expression without an operator. 
3. No two non-unary operators may appear consecutively without an operand between them. 
4. Operands are resolved first individually before being used in expressions. 
5. Operand data types should match the given syntax for each expression.
6. Nesting expressions are allowed, wherein parentheses must be balanced. Every opening parenthesis ( ( ) must have a matching closing parenthesis ( ) ). An empty pair of parentheses is not allowed.
7. Nested expressions are evaluated from the innermost to the outermost pair of parentheses.
8. An expression may only start with: (a) An operand, (b) An opening parenthesis, (c) A unary operator ( -, not ). 
9. An expression may only end with: (a) An operand, (b) A closing parenthesis.
10. When an expression combines different types of operands, it results in an error.
11. Each expression resolves to one primitive data type, which could either be piece, sip, flag, or chars.
12. Precedence Order for Evaluating Expressions:
   1. Recipe calls / Accessing
   2. Parentheses ( )
   3. Unary operators ( not, - )
   4. Multiplication, Division, Modulus ( *,  /,  % )
   5. Addition, Subtraction (+ , - )         
   6. Ordering ( < , <= , > , >= )
   7. Equality ( ==, != )
   8. Logical AND ( and )
   9. Logical OR ( or )
13. All expressions are left-associative, where expressions of the same precedence are evaluated from left to right, unless parentheses ( ) are used to override default precedence. 
14. Assignment operators ( =, +=, -=, *=, /=, %= ) are evaluated last and cannot be used more than once as it can only be used as the first operator in an assignment statement.
15. Expressions on their own are not valid statements.


Syntax: 
# Simple Expression
<operand>
<operand> <operator> <operand>;

2. Unary minus ( - ) sign negates a numeric literal only.
3. ( - ) must appear immediately before a numeric literal, with no whitespace.
4. The not operator negates a flag value. 
5. The not operator can only be used before an operand. The not operator may appear adjacent to other logical operators and itself.

-<numeric_literal>

Arithmetic Expressions
Rules:
1. Operators for arithmetic expressions include addition ( + ), subtraction ( - ), multiplication ( * ), division ( / ), and modulus ( % ).
2. Modulus ( % ) can only be used to operate on piece values.
3. Division between piece values produces an integer quotient, discarding the remainder value; exact decimal results only occur if the operands are of type sip, and zero or an expression evaluating to zero as the divisor results in an error. 
4. Two minus ( - ) symbols next to each other are only allowed if separated by a space or opening parenthesis ( ( ). 
5. Arithmetic expressions follow the ordered precedence for expressions.
6. Arithmetic operators do not operate on flag types and will result in an error.
7. Arithmetic operators subtraction ( - ), multiplication ( * ), division ( / ), and modulus ( % ), do not apply to chars types while addition ( + ) will only be utilized for chars concatenation.
8. Arithmetic expressions may also contain unary expressions with the unary minus ( - ) operator on piece and flag literals.

Logical Expressions
Rules:
1. Operators for logical expressions include AND ( and ), OR ( or ), and NOT ( not ). 
2. Logical expressions only operate on flag values and produce a flag result (up/down).          
3. The not operator can only be used before an operand, where the not operator may also appear adjacent to other logical operators and itself.
4. Logical operators must not attach to its operands and should be space or parenthesis separated.
5. and results in up only if both operands evaluate to up, else it results in down; or results in up if at least one operand evaluates to up, else it results in down; not negates a flag value.
6. Expressions may not begin or end with an and or or logical operator.
7. Logical Expressions are evaluated following the precedence order for expressions.

<flag_operand> or <flag_operand>

Relational Expressions
Rules: 
1. Relational expressions compare two values and produce a flag result ( up / down).
2. Relational operators include greater than ( > ), less than ( < ), greater than or equal to ( >= ), less than or equal to ( <= ), equal to ( == ),  and not equal to ( != ).
3. Comparisons are only allowed between piece with piece, sip with sip,  flag with flag, and  chars with chars. Any attempt to compare across incompatible types results in an error.
4. When comparing flag to flag values:
   1. Only ( == ) and ( != ) are allowed.
   2. Ordering relational operators ( <, >, <=, >= ) are not allowed.
5. When comparing chars to chars values:
1. Equality operators ( == ) and ( != ) is case sensitive and will evaluate if both chars contain the exact same length and characters. 
2. Ordering relational operators ( <, >, <=, >= ) are evaluated lexicographically.

	Array 
An array is a collection of values of the same data type, table prototype or another array that can be organized in multiple dimensions and accessed using an index. Values inside an array are called elements.


Rules in declaring an array:
1. When declaring an array, it must have the following in order:
   1. The data type or table prototype identifier
   2. Dimension square bracket pairs ( [ ] )

   5. A statement terminator ( ; )

4. Multiple array declarations in a single statement is valid. Each identifier must be separated by a comma ( , )
5. Multiple arrays in the same declaration statement will acquire the same array type.
6. The size of an array dimension is not explicitly declared.
7. An array has a dynamic size, allowing elements to be freely added or removed using built-in recipes.
8. An array may be a collection of a primitive data type: piece, sip, chars, or flag
9. An array may be a collection of a table when using a table prototype in its declaration
10. Arrays can be multidimensional. This means an array may also be a collection of arrays of primitive data, table, or another array.
11. Arrays that are not initialized will have an empty array as its value.        
12. Operators cannot be used between arrays except the assignment operator equal (=) only.
13. Arrays can be declared in the global scope and all global array declarations must be declared before the recipe definitions and the start recipe.

A table is a collection of values containing multiple fields based on a prototype. A field is an internal ingredient, array or another table that represents a unit of data within the table.


Rules in defining a table prototype
1. A table prototype can only be defined in global scope.
2. When defining a table prototype, it must have the following in order:

9. Fields cannot be re-declared within the same table prototype.
10. Fields identifiers will not cause a conflict outside of its table prototype.
11. A table prototype must contain 1 or more fields. Defining an empty table prototype will cause an error.
12. A table prototype cannot use itself directly as a type for its field. Only defined table prototypes before the new table definition can be used as a type for its values.
13. All defined table prototypes must have field declarations.
14. A table’s fields may contain the following values
   1. Primitive data types: piece, sip, chars, or flag
   2. Another table or array
15. A table’s field cannot contain another table prototype or recipe as a value.
16. Operators cannot be used between table prototypes.

platter
A platter is a block of code where it groups a sequence of executable statements and encloses it with curly brackets ( {} ).
Rules:
1. A platter starts with the opening curly bracket ( { ), followed by the local declarations (optional) and statements (optional), and ends with a closing curly bracket ( } ).
2. A platter may contain zero statements.
3. Each platter contains its own local scope. 
4. A platter cannot be used alone without a statement, clause, or recipe definition.
5. A platter has five variants, each designed to accommodate specific jump statements:
   1. platter - allows serve only.
   2. loop platter - allows all jump statements 
   3. menu check platter - allows stop and serve only.
   4. menu platter - allows only stop and serve and is exclusive to menu statements.
   5. menu loop platter - allows all jump statements and is exclusive to menu statements.
6. The Recipes and the Start Recipe must be paired with a platter.
7. A platter is paired with the following control-flow statements and clauses: (a) check, (b) alt, (c) instead, (d) menu, (e) pass, (f) repeat statement, (g) order.
Syntax:
# platter
{
     <local_decl/s>/none
     <statements/s>/none
}

     <statements_loop/s>/none

     <local_decl_menu/s>/none
     <statements_menu/s>/none
}


# menu_platter
{
     <choice_clause/s>/none
     <usual_clause>/none
}


# menu_loop_platter
{
     <choice_clause_loop/s>/none
     <usual_clause_loop>/none
}

	Recipes
Recipes are reusable platters, similar to subprograms, crafted to perform specific tasks. They help make code cleaner, more organized, and easier to maintain.


Rules in defining a recipe: 
1. A recipe is defined by creating (1) the recipe header, which consists of the keyword prepare to declare a recipe, the recipe serve type, the keyword of, the recipe identifier, the list of spices, and then (2) the platter.
2. The serve type indicates the data type, array type, or table prototype identifier that the recipe will serve after being called. Recipes can only have one serve type, which could either be piece, sip, flag, chars, array type, or table prototype. 
3. Recipes with the same identifier are not allowed. It must be a completely unique identifier within the global scope and should not have the same identifier with any globally declared ingredients, arrays, tables, or table prototype. Conflict with the same identifier on a local scope will only occur if the recipe with the same identifier is called in that local scope.
4. A recipe must be defined after the global declarations, if there are any present, and before the start recipe. Recipes cannot be defined before the global declarations. Recipes cannot be defined inside nor after the start recipe. 
5. Recipes cannot be forward-defined, the full definition must be written before calling it.
6. A recipe cannot be defined inside another recipe, nesting recipe definitions is not allowed.
7. There is no maximum limit to the number of recipes that may be declared in a single program. 
Syntax:
prepare <serve_type> of <id>(<spice1>,<spicen>/none){
     <local_decl/s>/none
     <statements/s>
}
	Where: 


<serve_type> = <piece/sip/flag/chars/id>
<spice> = <piece/sip/flag/chars/id> of <id>
<statements> = 
     <id_statements> /
     <built_in_rec_call> /
     <conditional_st> /
     <looping_st> /
     <jump_serve>
	

Rules for Spice:
1. A spice refers to the parameters that the recipe will require to be provided when calling it. These will serve as the input values of the recipe.
2. The list of spices shall be defined in the recipe header, and enclosed in a pair of parentheses following the ingredient, array, and table declaration rules respectively except that multiple spices cannot share a single declaration.         
3. The list of spices may be empty, indicated by an empty pair of parentheses ( () ). In such cases, the recipe performs its task without requiring input values.
4. The spice is also considered a local declaration of the recipe’s platter.
5. Multiple spices may exist in the list of spices, and are separated by a comma ( , ).
Syntax:
(<spice_type> of <id>, .../none) 
	Where:
<spice_type> = piece / sip / flag / chars / <array_type> / <table_prototype_id>
	



Rules for the Recipe Platter:
1. A recipe must contain exactly one top-level platter, in which the platter begins with ( { ) and ends with ( } ).         
2. A recipe platter follows the platter rules, but a recipe’s platter must always have at least one serve statement.
3. The recipe platter contains all the statements in the recipe, including the required serve statement. 


Syntax:
{
    <local_decl/s>/none
    <statements/s>
}
	Where: 
<local_decl> = piece        <piece_decl> /
chars        <chars_decl> /
sip        <sip_decl> /
flag        <flag_decl> /
id        <local_id_tail>


<statements> = 
     <id_statements> /
     <built-in_rec_call> /
     <conditional_st> /
     <looping_st> /
     <jump_serve>
	________________
Rules for the Serve Statement:
1. A serve statement must start with the keyword serve, a whitespace, followed by the value to be served by the recipe, and then it should end with a statement terminator ( ; ).
2. A serve statement can be used anywhere after the local declarations in a recipe platter to output a value and exit the recipe.
3. The type of the value to be served must match the serve type for the primitive data types.
4. The dimension or array type of the value to be served must match the serve type of the recipe for arrays.
5. The table prototype of the table value to be served must match the serve type of the recipe for tables. 
6. Recipes must contain at least one serve statement in the recipe platter; multiple serve statements may exist but only one will be executed and will serve a value when the recipe is called. A serve statement placed inside an unreachable branch, like by being locked behind a never-entered condition, does not satisfy this requirement.
7. An incomplete serve statement with a missing value is not allowed and will cause an error.
8. For recipes that do not serve a meaningful value, such as the bill recipe, use 0 as the serve value to fulfill the serve statement requirement, and in such cases, the recipe must be defined with piece as its serve type.
Syntax:
serve <value>;
	Where: 
<value> = <any_expr> /
     <ret_array> /
     [<notation_val>] 
	

________________
Rules for Flavor:
1. Flavors are the values passed in the recipe call. 
2. Flavors must match the quantity and type of the spices in the recipe definition and in the right order.
3. Recipes only allow passing by value, the flavor added in the recipe call acts as the initialization value of the declared spices. And the original references of the passed flavors are not going to be modified through this.


Rules for Recipe Calls:
1. Calling or invoking a recipe means executing its platter using a specific array of flavors that match the recipe's declared spices in the same quantity and order. Invocation must respect the recipe’s serve type.
2. The flavors are evaluated before the execution of the recipe.
3. To call a recipe, it must contain the recipe identifier, and the list of flavors to be passed, enclosed in parentheses ( () ). This list can be empty indicated by an empty pair of parentheses ( () ).
4. ​Recipes may call other recipes any number of times, with no imposed limit on the amount of calls within a recipe or the start recipe, however the called recipes must already be completely defined prior to the call.
5. User-defined recipes cannot be called in the global scope.
6. Recipes can be called in any local scope after the local scope’s local declarations as recipes are globally defined, but calling a recipe with the same identifier to a declared local ingredient, array, or table will cause an error.
7. Nested invocation of recipes is allowed and is evaluated from the inside out. Inner recipes are resolved first, and their results are passed as flavors to the outer recipe call. 
8. A recipe call used without a statement terminator is treated as an expression operand. It executes its defined statements and prioritizes the served value for use within the surrounding expression.
9. A recipe call used as a standalone statement must end with a statement terminator (;). In this form, the recipe is invoked for its behavior, and its served value is discarded unless explicitly handled.


________________


Syntax:
# recipe call
<recipe_id>(<flavor>, <flavor>, .../none)


# recipe call statement
<recipe_id>(<flavor>, <flavor>, .../none); 


# nested recipe call inside a recipe call statement
<recipe_id>(<recipe_call>);
<recipe_id>(<recipe_id>(<recipe_call>));
...
	Where: 
<flavor> = 
     <value>
	

Rules for Recursive Recipe Definition and Calls:
1. A recipe can call itself within its own platter.
2. Each recursive call must supply the correct number and types of flavors as defined in the recipe’s spice list.
3. Recursive recipes must still contain exactly one top-level platter and at least one serve statement.


Syntax: 
prepare <serve_type> of <recipe_id>(<spice_list>){
     <recipe_id>(<flavor>, <flavor>, ...);      # recursive call
     serve <value>;
}

Built-in Recipes
Built-in-recipes are the predefined recipes that come with Platter. They perform special tasks and follow strict syntax rules that cannot be redefined.


Type Casting


topiece
A topiece recipe type casts sip, chars, or flag values into a piece type.


Rules:
1. A topiece recipe only accepts one piece, sip, chars, or flag value.
2. Casting piece values to piece does nothing and serves the original value.
3. Casting sip values to piece truncates the fractional part, dropping everything after the decimal point then the decimal point itself. 
4. Casting chars values to piece extracts the piece value contained within the chars. The piece value is detected within the chars value by reading each character if it follows a piece literal format. 
5. If the chars contains a character that resembles a sip literal, it also truncates the fractional part and serves the non-fractional part as a piece.
6. If the chars does not contain a valid numeric representation following the piece or sip literal rules, it will serve an error.
7. Leading zeroes are dropped when casted with topiece, except if the value consists entirely of zeros in which case it collapses into a single 0.
8. Casting flag values to piece serves 0 for down and 1 for up.
9. Any other flavor type will result in an error.
10. The topiece recipe serves a value of type piece based on the passed flavor.


Syntax:
topiece(<any_expr>)

tosip 
A tosip recipe type casts piece, chars, or flag values into a sip type. 


Rules:
1. A tosip recipe only accepts one piece, sip, chars, or flag value.
2. Casting piece values to sip will add a decimal point and one trailing zero at its decimal place.
3. Casting sip values to sip does nothing and serves the original value.
4. Casting chars values to sip extracts the sip value contained within the chars. The sip value is detected within the chars value by reading each character if it follows a sip literal format.
5. If the chars contains a character that resembles a piece literal, it also adds a decimal point and one trailing zero at its decimal place. It then serves the digits before the period, the period, and the digits after the period as a single sip value.
6. If the chars does not contain a valid numeric representation following the piece or sip literal rules, it will serve an error.
7. Leading zeroes in the non-fractional digits are dropped when casted with tosip, except if that part consists entirely of zeros in which case it collapses into a single 0 followed by the decimal value.
8. Leading zeroes are dropped when casted with tosip, except if the non-fractional digits contain only one zero digit.
9. Casting flag values to sip serves 0.0 for down and 1.0 for up.
10. Any other flavor type will result in an error.
11. The tosip recipe serves a value of type sip based on the passed flavor.

Input and Output Recipes
Built-in recipes handle the processing of data received from external sources using the take recipe, and produce output or results through the bill recipe.


take
        A take recipe allows the program to accept and store user input.


Rules:
1. The take recipe awaits for user input and halts the program until it encounters a chars input delimited by a newline.
2. The user input is stored as a chars literal omitting the last newline used to delimit and end the input.
3. If the user input starts with a newline, it will be omitted as well and the take recipe serves an empty chars.
4. When assigning a take recipe to another data type than chars, it must be explicitly typecasted. 
5. If the user inputted characters of unsupported ascii characters based on the accepted chars literal rules, the take recipe will only serve all characters that are supported.
6. The take recipe serves a value of type chars.


Syntax:
take()

Local Declaration
Ingredients 
Ingredients follow the same global declaration rules in its local declaration with some additional rules:
1. All local ingredient declarations must be declared before any other statement within its scope.
2. It follows the same syntax of global ingredient declarations.
3. If an ingredient is declared both globally and locally, the local declaration will override the global one for its local scope and its value will be the one accessed and modified instead. 
4. For this reason, a global and a local ingredient can have the same identifier and the global ingredient will only be accessible if the local scope did not use the same identifier for any ingredient, array, or table.
5. Ingredients declared in a local scope can only be accessed within its said local scope. Trying to access an ingredient outside of its scope returns an error.
6. Ingredients  follow the same global initialization rules for local ingredient initialization.
7. The syntax for both the declaration and initialization follows the same one for the global ingredient declarations.

Arrays and Tables 
Arrays and tables follow the same global declaration rules in its local declaration with some additional rules:


1. All local arrays and table declarations must be declared before any other statement within its scope.
2. Table prototypes cannot be defined in a local scope. It can only be defined in the global scope.
3. Arrays and tables must not have the same identifier name as table prototypes, a recipe, or an ingredient in global scope or within the same local scope.
4. If an array or table is declared both globally and locally, the local declaration will override the global one for its local scope. The compiler will always use the version that is closest to the point of reference.
5. Arrays and tables declared in a local scope can only be accessed within that same local scope.


Rules in re-assigning and accessing arrays and tables:
1. Local reassignment can modify global arrays or tables only if no array, table, or ingredient with the same identifier exists in the local scope.
2. Reassigning a new value to an array or table replaces the old value entirely. The old value may come from a broader scope (e.g., a global declaration) or from a prior assignment within the same scope.
3. Array and table reassignment can only occur within a platter and is not permitted at the top level.

Statements
A statement is a complete instruction that specifies an action for the program to perform. The following are defined as statements: (a) local assignment, (b) standalone recipe call, (c) conditional: check and menu, (d) looping: pass, repeat, and order-repeat, and (e) jump: next, stop, and serve.

Rules in reassigning ingredients: 
1. Ingredients are mutable containers of data and can be reassigned any value as long as it is accessible within the current scope.
2. Reassigning new values to an ingredient causes the old value to be replaced by the new value. Old value refers to values assigned in a broader scope or values assigned to the ingredient in the earlier parts of the code.
3. Local reassignment can modify global ingredients if no ingredient, array, or table has the same identifier in the local scope. Reassigning a global ingredient value will modify the global scope value of the ingredient, and the new value is not limited to the local scope in which it was assigned.
4. Reassigning an ingredient’s value by accessing it from a deeper nested platter level causes it to behave like a local scope accessing a global scope variable in which the global scope variable is modified and accessed from the deeper local scope.
5. Value reassignment is done by using the ingredient identifier, followed by an assignment operator, and the valid expression, both the ingredient and the resolved expression should be of the same data type. 
6. Only a single ingredient reassignment is allowed, multiple ingredient reassignment is not allowed in a single statement and each reassignment should be completed with a statement terminator ( ; ).

A conditional statement is a control-flow statement that allows a program to make decisions based on whether a given condition is up (true) or down (false).


check
A decision-making control structure that allows the program to choose one block of code from multiple alternatives, based on which condition is true.


Rules:
1. A check statement must start with the reserved word check, followed by its condition enclosed in parentheses ( () ), and then a platter.
   1. When creating a check statement with an instead clause, the platter of the check statement must be followed by an instead reserved word and then end with a platter. 
   2. When creating a check statement with an alt clause, the platter of the check statement must be followed by an alt reserved word, followed by its condition enclosed in parentheses ( () ), and end with a platter.
   3. When creating a check statement with an alt and instead clause, the platter of the check statement must be followed by an alt reserved word, followed by its condition enclosed in parentheses ( () ), then a platter, and then followed by an instead reserved word, and end with a platter.
2. A check statement may include an alt clause, an instead clause, or both.
3. A check statement can be used without an alt and instead clauses.
4. A check statement and each alt clause must have exactly one condition and that condition must evaluate to a flag value.
5. Zero or more alt clauses are allowed. If present, it must be associated with a preceding check statement.
6. An instead clause is optional. If present, it must appear at most once and it must be associated with a preceding check statement. If one or more alt clauses exists within the same check statement, the instead clause must appear after all of them.
7. The execution of a check statement proceeds as follows: 
   1. The check condition is evaluated first.
      1. If condition evaluates to up, the platter of check executes, and all alt and instead clauses are skipped, if any.
      2. If condition evaluates to down, the alt clause/s, if any, are evaluated sequentially in the order they appear.
   2. When an alt condition evaluates to up, its platter executes, and the remaining alt clause/s and instead clause, if any, are skipped.
   3. If no alt condition evaluates to up, and an instead clause exists, the instead platter executes.
   4. If no alt conditions are up and no instead clause exists, no platter is executed.

menu
A multi-way branching control structure that allows the program to choose one choice clause to execute based on the value of the menu expression.
Rules:
1. A menu statement must start with the reserved word menu followed by an opening parenthesis ( ( ), followed by a condition, followed by a closing parenthesis ( ) ), and then end with a platter.
   1. When creating a choice clause inside the platter of the menu, it must start with the reserved word choice, followed by a choice value, then a colon ( : ). It may then contain zero or more statements.
   2. When creating a usual clause inside the platter of the menu, it must start with the reserved word usual, then a colon ( : ). It may then contain zero or more statements..
2. A menu statement may contain a choice clause or a usual clause. 
3. A menu statement must have exactly one condition and that condition must evaluate to a flag value.
4. The menu statement only accepts piece values and chars values as its condition.
5. Zero or more choice clauses are allowed. If present, it must be associated with a preceding menu statement.
6. Every choice must have exactly one choice value and it must be unique within the same menu statement.
7. All choices only accept piece and chars literals as their choice value.
8. Each choice clause may optionally end with a stop statement. If a stop statement is omitted for a certain choice, the execution continues to the next choice until a stop statement is encountered or until the menu statement terminates.
9. A usual clause is optional. If present, it must appear at most once and it must be associated with a preceding menu statement. If one or more choice clauses exists within the same menu statement, the usual clause must appear after all of them.
10. A usual clause may optionally end with a stop statement.
11. The execution of a menu statement proceeds as follows:
   1. The condition is evaluated first.
   2. The result is compared to each choice value in order.
   3. When a matching choice value is found, that choice clause is executed.
      1. Execution continues until a stop statement is encountered.
      2. If a stop statement is encountered, the program exits the menu statement immediately after that choice clause;
      3. Otherwise, execution falls through the next choice clause, until a stop statement is encountered.
   4. If no choice values match, the usual clause is executed.
   5. If no choice values match and no usual clause exists, then the entire menu statement will be executed until the end of its platter.

A looping statement is a control-flow statement that allows a program to repeat one or more instructions as long as a specified condition is true.
pass

Rules:
1. A pass statement must start with the pass reserved word followed by an opening parenthesis ( ( ), followed by the following: its initialization, update, and condition, followed by a closing parenthesis ( ) ), and end with a loop platter. 
2. The initialization, update, and condition must be separated with a semicolon (;).
3. The initialization, update, and condition must each contain exactly one:
   1. Ingredient to assign or reassign for initialization. The assigned ingredient is limited to type piece. When assigning a value, it must resolve to a piece value.

   1. The initialization is executed once.
   2. The condition is evaluated.

      2. If the condition evaluates to down, then the loop terminates.

The Start Recipe
The start recipe is the designated entry point of every Platter program. It represents the main sequence of instructions that is executed when the program runs.


Rules:
1. In declaring the start recipe, it must have:
   1. The keyword start.
   2. An empty pair of parenthesis ( ( ) ), indicating empty spices.
   3. A platter which defines the main program body, following the platter rules.
2. Multiple definitions of the start recipe in the same program is not allowed.
3. Every program must have exactly one start recipe declaration
4. The start recipe must be defined at the program’s top level and is never nested inside a recipe or platter.
5. Within the start recipe, the following are accessible:
   1. Global ingredients, arrays, table prototype and tables.
   2. Local ingredients, arrays, table prototype and tables within its scope.
   3. Recipes defined within the program.
6. A start recipe must not have any spices inside its header.
7. The start recipe can have more than one serve statement that can be used to return an error.
8. The program ends without an error when the start recipe has finished executing all the statements in its platter, which will implicitly serve an exit code of 0. If a serve statement is encountered during execution inside the start recipe, the program exits with the serve value as the exit code.
9. The start recipe implicitly serves 0 when the program is executed successfully without errors.
10. The start recipe cannot be called in the program.
11. The start recipe is defined only after the global declarations, and recipes in the program.
